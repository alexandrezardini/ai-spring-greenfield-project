# Liquibase Migration Rules

## Schema Management

- All database schema changes must be performed through **Liquibase changelogs**.
- Never perform manual DDL operations on the database.
- Each logical change (e.g., creating a table, adding a column) must be in a separate `changeSet`.

## ChangeSet Conventions

- **ID:** Use a sequential number or a descriptive slug (e.g., `create-user-table`).
- **Author:** Use your identifier or "IA" for changes generated by the model.
- **Fail-fast:** Include `preConditions` where appropriate to ensure the state is as expected before applying the change.
- **Rollback:** Always provide a `<rollback>` section for every `changeSet` to allow for reverting changes.

## Best Practices

- Use **YAML**, **XML**, or **JSON** for changelogs to maintain database independence, or use **formatted SQL** if complex features are needed.
- **Master Changelog:** All individual changelog files must be included in the `db.changelog-master.yaml`.
- **Naming Files:** Use a timestamp or a sequential prefix for new changelog files: `001-initial-schema.yaml`, `002-add-video-table.yaml`.
- **Avoid Modifying Existing ChangeSets:** If a mistake was made in a previously deployed `changeSet`, create a new one to fix it. Do not change the original as it will cause checksum validation failures.

## Example (YAML)

```yaml
databaseChangeLog:
  - changeSet:
      id: 20240421-create-users
      author: junie
      changes:
        - createTable:
            tableName: users
            columns:
              - column:
                  name: id
                  type: UUID
                  constraints:
                    primaryKey: true
                    nullable: false
              - column:
                  name: username
                  type: VARCHAR(50)
                  constraints:
                    nullable: false
                    unique: true
      rollback:
        - dropTable:
            tableName: users
```

## The Rule

- Every change to an Entity's persistent fields must be accompanied by a corresponding Liquibase `changeSet`.
- Verify the migration by starting the application and checking the database schema.
- Keep `changeSets` small and focused.
